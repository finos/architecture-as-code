# Day 23: Contribute to the CALM Community

## Overview
Give back to the CALM open-source community by contributing documentation, reporting issues, or helping others.

## Objective and Rationale
- **Objective:** Make a meaningful contribution to the CALM community - report an issue, improve documentation, answer a question, or share your learnings
- **Rationale:** Open source thrives on contribution. Giving back strengthens the ecosystem, helps others on their CALM journey, and deepens your own understanding. Build your open-source portfolio while improving CALM for everyone.

## Requirements

### 1. Review Contribution Options

Choose one or more ways to contribute:

**Option A: Report an Issue**
- Found a bug in CLI?
- Schema documentation unclear?
- Unexpected validation behavior?

**Option B: Improve Documentation**
- Fix typos or unclear explanations
- Add examples to schema docs
- Improve README sections

**Option C: Answer Questions**
- Help in GitHub Discussions
- Answer questions in FINOS Slack
- Share solutions to common problems

**Option D: Share Your Journey**
- Write blog post about CALM
- Create tutorial or video
- Present at meetup/conference

**Option E: Contribute Code**
- Fix a bug
- Add a feature
- Improve test coverage

**Option F: Share Examples**
- Submit example architecture
- Create template bundle
- Share pattern

### 4. Make Your Contribution

#### Option A: Report an Issue

**Steps:**
1. Reproduce the issue clearly
2. Check if it's already reported
3. Create new issue on GitHub
4. Provide clear description, steps to reproduce, expected vs. actual behavior
5. Include CALM CLI version, OS, sample files if relevant

**Template:**
```markdown
## Description
[Clear description of the issue]

## Steps to Reproduce
1. [Step 1]
2. [Step 2]
3. [Step 3]

## Expected Behavior
[What should happen]

## Actual Behavior
[What actually happens]

## Environment
- CALM CLI version: [From `calm --version`]
- OS: [Linux/Mac/Windows]
- Node version: [From `node --version`]

## Sample Files
[Attach or link to files that reproduce the issue]
```

#### Option B: Improve Documentation

**Steps:**
1. Fork the repository
2. Find documentation to improve (README, schema docs, etc.)
3. Make improvements
4. Submit pull request

- Add clarifying examples
- Improve formatting
- Add missing links
- Clarify confusing sections

[Link to PR]

## Tips
- Start small - even fixing a typo helps
- Quality over quantity - one good contribution beats many rushed ones
- Be respectful and follow community guidelines
- Ask questions if unclear - maintainers want to help
- Document your contribution process for others
- Consider how your contribution helps future users
- Open source is collaborative - expect feedback
- Every contribution counts, no matter how small
