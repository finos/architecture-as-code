## Summary

Phase 08.1 delivers a cohesive set of five features that all share the same underlying infrastructure: CALM 1.2 controls support. The work layers cleanly on top of the existing extension pack, validation, and properties panel systems. Roughly 70% of the work follows well-established patterns already present in the codebase; the remaining 30% is genuinely new territory (GovernancePanel, AIGF data package, template picker modal).

The two pre-existing template files (`fluxnova-platform.json`, `fluxnova-kyc-onboarding.json`) already contain CALM 1.2 controls in the correct format — they are functional now but will fail to round-trip cleanly until `CalmNode.controls` is added to `calm-core/types.ts`. This type gap is the critical prerequisite for everything else. Once controls exist in the type system, every other feature can be implemented independently.

The AIGF catalogue (`docs/AIGF_CATALOGUE.json`) is fully pre-structured with 23 risks and 23 mitigations in the correct TypeScript-mappable shape. No transformation is needed beyond importing and typing it.

**Primary recommendation:** Sequence work in this order — (1) calm-core type additions, (2) FluxNova pack registration, (3) template system, (4) controls in properties panel, (5) AIGF data package and governance panel, (6) validation rules, (7) decorator export. Each step is a releasable increment.

### Pattern 2: CALM Types Extension (CTRL-01, CTRL-03)
**What:** Extend `CalmNode` and `CalmRelationship` in calm-core with controls, data-classification, metadata, and add decorator/evidence types.
**Critical:** `CalmControlRequirement` follows the CALM 1.2 `oneOf` requirement: either `config-url` OR `config` but NOT both. The `requirement-url` is mandatory. However — the existing templates use requirements with ONLY `requirement-url` (no config), which is a valid subset.
**Example:**
```typescript
// packages/calm-core/src/types.ts
interface CalmControlRequirement {
  'requirement-url': string;
  'config-url'?: string;
  config?: Record<string, unknown>;
}

interface CalmControl {
  description: string;
  requirements: CalmControlRequirement[];
}

type CalmControls = Record<string, CalmControl>;

// Add to CalmNode:
controls?: CalmControls;
'data-classification'?: string;
metadata?: Record<string, unknown>;

// Add to CalmRelationship:
controls?: CalmControls;
metadata?: Record<string, unknown>;

// New top-level types:
interface CalmDecorator {
  'unique-id': string;
  type: string;
  target: string[];
  'applies-to': string[];
  data: Record<string, unknown>;
}

interface CalmEvidence {
  'unique-id': string;
  'evidence-paths': string[];
  'control-config-url': string;
}

// Extend CalmArchitecture:
interface CalmArchitecture {
  nodes: CalmNode[];
  relationships: CalmRelationship[];
  decorators?: CalmDecorator[];   // ADD
}
```

### Pattern 3: Template System (TMPL-01, TMPL-02, TMPL-03)
**What:** Static JSON template files loaded via import registry, with a full-screen modal picker.
**Template format:** Standard `CalmArchitecture` fields + `_template` metadata block (stripped on export).
**Example:**
```typescript
// apps/studio/src/lib/templates/registry.ts
import type { CalmArchitecture } from '@calmstudio/calm-core';

interface CalmTemplate extends CalmArchitecture {
  _template: {
    id: string;
    name: string;
    description: string;
    category: string;   // 'fluxnova' | 'ai-governance' | 'general'
    tags: string[];
    version: string;
    author: string;
    sourceRef?: string;
  };
}

const templates = new Map<string, CalmTemplate>();

export function registerTemplate(t: CalmTemplate): void {
  templates.set(t._template.id, t);
}

export function loadTemplate(id: string): CalmArchitecture {
  const t = templates.get(id);
  if (!t) throw new Error(`Template ${id} not found`);
  // Strip _template metadata — CALM JSON must not contain it
  const { _template, ...arch } = t;
  return arch;
}

export function getTemplatesByCategory(category: string): CalmTemplate[] {
  return [...templates.values()].filter((t) => t._template.category === category);
}

export function getAllCategories(): string[] {
  return [...new Set([...templates.values()].map((t) => t._template.category))];
}
```

### Pattern 5: Governance Store (AIGF-02, AIGF-03)
**What:** Module-level `$state` reactive store that computes governance score from the canonical model.
**Pattern derived from:** `validation.svelte.ts` store — same module-level $state singleton pattern.
**Example:**
```typescript
// apps/studio/src/lib/stores/governance.svelte.ts
import { getModel } from './calmModel.svelte';
import { getAIGFForNodeType } from '@calmstudio/calm-core/aigf';

let selectedNodeGovernance = $state<{ risks: AIGFRisk[]; mitigations: AIGFMitigation[] }>({ risks: [], mitigations: [] });
let architectureScore = $state<number | null>(null);
let hasAINodes = $state(false);

export function refreshGovernance(): void {
  const model = getModel();
  const aiNodes = model.nodes.filter(isAINode);
  hasAINodes = aiNodes.length > 0;
  architectureScore = computeGovernanceScore(model, aiNodes);
}

export function updateSelectedNodeGovernance(nodeType: string | null): void {
  if (!nodeType) { selectedNodeGovernance = { risks: [], mitigations: [] }; return; }
  selectedNodeGovernance = getAIGFForNodeType(nodeType);
}
```
**Key:** Score updates on control apply/remove, NOT on explicit Validate trigger. Call `refreshGovernance()` from `onmutate` callbacks in properties panel.

### Pattern 6: AIGF Validation Rules (AIGF-05)
**What:** Functions that examine a `CalmArchitecture` and return `ValidationIssue[]`. Aggregated into the existing `runValidation()` call.
**Example:**
```typescript
// apps/studio/src/lib/validation/aigf-rules.ts
import type { CalmArchitecture } from '@calmstudio/calm-core';
import type { ValidationIssue } from '$lib/stores/validation.svelte';

export function runAIGFRules(arch: CalmArchitecture): ValidationIssue[] {
  const issues: ValidationIssue[] = [];
  for (const node of arch.nodes) {
    if (isAINode(node) && !hasAnyAIGFControl(node)) {
      issues.push({
        severity: 'warning',
        message: `AI node "${node.name}" has no AIGF governance controls applied`,
        nodeId: node['unique-id'],
        ruleId: 'aigf-001',
      });
    }
    // ... rules aigf-002 through aigf-010
  }
  return issues;
}
```
**Integration:** In `validation.svelte.ts`, change `runValidation()` to:
```typescript
import { runAIGFRules } from '$lib/validation/aigf-rules';
export function runValidation(): void {
  const model = getModel();
  const structural = validateCalmArchitecture(model);
  const aigf = runAIGFRules(model);
  issues = [...structural, ...aigf];
  panelOpen = true;
}
```

### Anti-Patterns to Avoid
- **Importing .svelte.ts into validation rules:** `aigf-rules.ts` must NOT import from `.svelte.ts` files — it must be pure TypeScript for vitest testability. Follow the `projection.ts` precedent.
- **Auto-switching to Governance tab on node select:** Locked out by user preference. Badge-only notification is the correct signal.
- **Embedding _template in exported CALM JSON:** Strip in the export function, not in the template loader (templates that are re-imported should retain metadata for re-use).
- **Computing governance score inside a Svelte component:** Compute in the `governance.svelte.ts` store; components only read state. Follows the validation store pattern.
- **Adding AIGF risks as `risks` property on CalmNode:** The CALM spec does not yet have a `risks` property on nodes. AIGF risks are computed dynamically at design-time by CalmStudio, not stored in the CALM JSON. Only _mitigations_ are stored (as controls).
- **CalmArchitecture type narrowing for controls roundtrip:** `updateNodeProperty(id, 'controls', value)` already works generically. No special mutation function needed for controls — they are just another node property key.

## Common Pitfalls

### Pitfall 1: Controls OneOf Constraint
**What goes wrong:** CALM 1.2 `control-detail` schema uses `oneOf` — a requirement must have EITHER `config-url` OR `config`, not both, not neither (other than `requirement-url`). However, the two existing templates use requirements with ONLY `requirement-url` and no config at all.
**Why it happens:** The schema `oneOf` technically requires one branch to be satisfied, but the existing templates pass validation. This suggests the validation is lenient or the oneOf is optional in practice.
**How to avoid:** When implementing the `CalmControlRequirement` TypeScript type, make both `config-url` and `config` optional (matching the `?` optional pattern). Don't add runtime validation that rejects requirements with neither config field — the existing templates would break.
**Warning signs:** Import of existing template fails type check after adding strict oneOf enforcement.

### Pitfall 2: Governance Score Reactivity Timing
**What goes wrong:** Governance score shown in toolbar badge might be stale if it's computed lazily or only on explicit governance panel open.
**Why it happens:** The score must update immediately when a control is applied/removed via "Apply" button, which happens in GovernancePanel — not in the properties panel's `onmutate` callback.
**How to avoid:** Call `refreshGovernance()` both from `onmutate` in properties panel AND from within the governance panel after applying a mitigation. The store pattern makes this safe (idempotent recompute).

### Pitfall 3: Template Loading vs Import
**What goes wrong:** Loading a template onto a canvas with dirty state, or loading a template that includes `fluxnova:` node types before the FluxNova pack is registered.
**Why it happens:** Template load calls `applyFromJson()` which projects to canvas immediately. If packs aren't registered, `resolvePackNode` returns null and nodes render as GenericNode.
**How to avoid:** `initAllPacks()` is already called at module level in `+page.svelte` (belt-and-suspenders pattern from Phase 7), so FluxNova pack will be registered before template load. Dirty-state confirmation must happen before `loadTemplate()` is called — not after.

### Pitfall 4: Data Classification Badge Clashes with Node Layout
**What goes wrong:** Adding a `data-classification` badge below the node name in canvas node components may cause layout overflow in small/compact node sizes, or may conflict with dark mode colors.
**Why it happens:** Canvas nodes have fixed-height rendering. Adding a badge element changes the effective height.
**How to avoid:** Render the badge as an absolutely-positioned element outside the normal node flow, or add padding compensation. Follow the validation badge pattern (already uses absolute positioning for error/warning counts). Use CSS variables for colors (`--color-error`, `--color-warning`, etc.) rather than hardcoded hex values.

### Pitfall 5: AIGF Validation Rules Import Path
**What goes wrong:** `aigf-rules.ts` importing from `$lib/stores/validation.svelte` (for the `ValidationIssue` type) creates a Svelte rune context issue in vitest.
**Why it happens:** `.svelte.ts` files use Svelte runes which require a Svelte compilation step that vitest's plain TypeScript runner doesn't provide.
**How to avoid:** Define `ValidationIssue` in `packages/calm-core/src/validation.ts` (pure TypeScript, already the source) and import from there. The existing `validation.svelte.ts` already does `export type { ValidationIssue }` as a re-export — `aigf-rules.ts` should import directly from calm-core, not from the store.

### Pitfall 6: Right Sidebar Tab Width
**What goes wrong:** Converting PropertiesPanel from single content to tabbed layout changes the minimum width of the right pane, which might push the canvas area too narrow.
**Why it happens:** Tab bar adds height at top and may add minimum width requirements. The paneforge Pane doesn't know about the new minimum.
**How to avoid:** Keep the tab bar compact (same 36px height as toolbar). The "Properties" and "Governance" labels are short enough to fit the existing panel width. Test at minimum pane width.

### Pitfall 7: Decorator on Export Requires Filename
**What goes wrong:** The CALM 1.2 decorator schema requires `target` to be a non-empty array of file paths or URLs. When the user hasn't saved the file yet, there is no filename.
**Why it happens:** Export function needs `target: ["<filename>"]` for the decorator. If `getFileName()` returns null, the decorator has no valid target.
**How to avoid:** Use `target: ["architecture.json"]` as a fallback when no filename is available. The decorator's primary purpose is conveying governance metadata; the target is informational for CalmGuard.

### Pitfall 8: Template Registry Init Order
**What goes wrong:** `getTemplatesByCategory()` returns empty if templates aren't registered before the TemplatePicker opens.
**Why it happens:** Templates need to be registered at module load time, similar to packs. If registration happens lazily (e.g., on first picker open), the first render shows nothing.
**How to avoid:** Register all templates at module import time in `registry.ts` itself (self-registering pattern), or register in `+page.svelte` at module level alongside `initAllPacks()`.

```typescript
// Derived from PRD B2.3
function computeGovernanceScore(arch: CalmArchitecture, aiNodes: CalmNode[]): number {
  if (aiNodes.length === 0) return 100;
  let totalRecommended = 0;
  let totalApplied = 0;
  for (const node of aiNodes) {
    const { mitigations } = getAIGFForNodeType(node['node-type']);
    totalRecommended += mitigations.length;
    for (const m of mitigations) {
      if (node.controls?.[m.calmControlKey]) totalApplied++;
    }
  }
  if (totalRecommended === 0) return 100;
  return Math.round((totalApplied / totalRecommended) * 100);
}
```

## Open Questions

1. **JSON import vs inline data for AIGF catalogue**
   - What we know: `docs/AIGF_CATALOGUE.json` is 53KB. TypeScript supports JSON imports with `resolveJsonModule` or import assertions.
   - What's unclear: Whether `packages/calm-core` has `resolveJsonModule: true` in its tsconfig.
   - Recommendation: Check `packages/calm-core/tsconfig.json`. If not enabled, inline the data as a typed TypeScript constant to avoid tsconfig changes and ensure tree-shaking.

2. **Canvas node component modifications for data-classification badge**
   - What we know: 9 core CALM node components + GenericNode + ExtensionNode all render node content. The badge needs to appear on nodes that have `data-classification` in their CALM data.
   - What's unclear: Whether `data-classification` from `CalmNode` gets projected into `node.data` via `projection.ts` / `calmToFlow`.
   - Recommendation: Check `projection.ts`'s `calmToFlow` function to confirm all CalmNode fields are passed into `node.data`. If not, add `data-classification` to the projection. The badge should be added to `ExtensionNode.svelte` (which handles all pack nodes including FluxNova) and `GenericNode.svelte` (fallback), rather than all 9 core node components.

3. **GovernancePanel selected node access**
   - What we know: `PropertiesPanel` receives `selectedNode` and `selectedEdge` as props from `+page.svelte`. The governance panel needs the selected node's `calmType` to look up AIGF mappings.
   - What's unclear: Whether `node.data.calmType` is sufficient or if the full `CalmNode` from `getModel()` is needed for controls check.
   - Recommendation: For the "is mitigation applied" check, `getModel()` gives the authoritative controls data. GovernancePanel should call `getModel().nodes.find(n => n['unique-id'] === selectedNode.data.calmId)` to get current controls — not rely on `node.data` which may not include controls if projection doesn't pass them through.
