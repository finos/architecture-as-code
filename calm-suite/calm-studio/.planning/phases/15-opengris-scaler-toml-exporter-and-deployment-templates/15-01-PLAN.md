Node-to-TOML-Section Mapping (from RESEARCH.md):
| CALM Node Type | TOML Section | Key Metadata |
|---|---|---|
| `opengris:scheduler` | `[scheduler]` | `scheduler_address` (default: `tcp://127.0.0.1:8516`) |
| `opengris:cluster` | `[cluster]` | `cluster_name` (default: `scaler-cluster`) |
| `opengris:worker` | (count contributes to manager's `max_workers`) | `worker_count`, `worker_capabilities` |
| `opengris:worker-manager` (native) | `[native_worker_manager]` | `manager_type=native`, `priority` |
| `opengris:worker-manager` (ecs) | `[ecs_worker_manager]` | `manager_type=ecs`, `ecs_cluster`, `ecs_task_definition` |
| `opengris:worker-manager` (symphony) | `[symphony_worker_manager]` | `manager_type=symphony` |
| `opengris:worker-manager` (aws_hpc) | `[aws_hpc_worker_manager]` | `manager_type=aws_hpc`, `aws_region` |
| `opengris:object-storage` | `[object_storage_server]` | `storage_address` (default: `tcp://127.0.0.1:8517`) |
| `opengris:client` | contributes `client_address` to output | `client_address` |
| `opengris:task-graph` | `[top]` | `max_parallel_tasks` (default: `4`) |
| `opengris:parallel-function` | contributes to `[top]` | `function_name` |

Known integer keys (emit unquoted in TOML): `worker_count`, `max_parallel_tasks`, `priority`, `port`, `max_workers`
Known boolean keys: (none currently)
All other metadata values: emit as quoted strings

TOML key collision pitfall: If two worker-managers share the same `manager_type`, append `_2` suffix to the second section header.

Address Auto-Derivation Algorithm (from RESEARCH.md, per locked CONTEXT.md decision):
1. Collect all `connects` relationships where `source` or `destination` matches the node.
2. If the peer node has an explicit address, derive the current node's address by incrementing the port by 1 (scheduler on 8516 -> object-storage on 8517).
3. Fall back to static defaults if no connected peer has an address.

  <implementation>
    Create `apps/studio/src/lib/io/scalerToml.ts` exporting `buildScalerToml(arch: CalmArchitecture): string`.

    Implementation approach:
    1. Filter `arch.nodes` to only `opengris:*` nodes
    2. Group by node type prefix
    3. Build each section using typed helper functions (one per TOML section type)
    4. Handle waterfall: collect worker-managers, sort by priority, emit section names in scheduler
    5. Handle type coercion: maintain a `Set` of known integer keys, emit unquoted for those
    6. Handle worker count: count `opengris:worker` nodes, use as default `max_workers`
    7. Handle duplicate manager_type: track seen types, append `_2` suffix on collision
    8. **Handle address auto-derivation from topology (per locked CONTEXT.md decision):**
       - Accept `arch.relationships` alongside `arch.nodes`
       - For each node missing an address-type metadata key (`scheduler_address`, `storage_address`, `client_address`):
         a. Find `connects` relationships where the node is source or destination
         b. Find the peer node and check if the peer has an explicit address in customMetadata
         c. If peer has address, parse the `tcp://host:port` format, increment port by 1 for the current node
         d. If no peer has address, fall back to static defaults
       - Port increment strategy: scheduler_address port + 1 = storage_address, scheduler_address port + 2 = client_address
       - This keeps the algorithm O(n) over relationships and deterministic
    9. Join sections with blank line separators, prepend header comment

    Anti-patterns to avoid:
    - Do NOT mutate `arch.nodes` or `arch.relationships` — read-only
    - Do NOT emit sections for non-opengris node types
    - Do NOT use a TOML library — hand-craft string (flat format, no nested tables)
    - Do NOT hardcode waterfall order — derive from priority metadata
    - Do NOT ignore `arch.relationships` — they are needed for address auto-derivation
  </implementation>
