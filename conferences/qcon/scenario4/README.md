# Scenario 4: Scaling Deployments and Operational Change

Demonstrates how a **platform-wide change** (adding resource limits to all workloads) is rolled out automatically by releasing a new bundle version — without any team-owned architecture files changing.

## Prerequisite

Scenario 3 cluster (`minikube --profile secure`) must be running with services deployed.

## Run the Demo

```bash
./demo.sh
```

## What the Demo Shows

### The Problem

- Inspects running pods from Scenario 3
- Confirms that no CPU or memory limits are defined
- Highlights the risk: runaway pods can exhaust cluster resources

### Platform Team Action (bundle-v1 → bundle-v2)

- Shows the diff between `bundle-v1` and `bundle-v2`
- `bundle-v2` adds platform-standard resource limits to deployment templates:
  - CPU: request `250m`, limit `500m`
  - Memory: request `128Mi`, limit `256Mi`

### Team Regenerates — Same Architecture, New Bundle

```bash
cd calm && calm template \
  --architecture trades-api-and-mcp-conforming.architecture.json \
  --bundle ../bundle-v2 \
  --output ../generated
```

The architecture file is **unchanged**. Only the `--bundle` argument switches from `bundle-v1` to `bundle-v2`.

### Deploy and Verify

- Applies the new manifests with `kubectl apply`
- Restarts the MCP deployment to pick up the changes
- Confirms resource limits are now enforced via `kubectl describe pod`

## Key Insight

> **Architecture = WHAT** (the system design — team-owned, stable)  
> **Bundle = HOW** (the platform's deployment opinion — platform-owned, evolves independently)

This is the log4j analogy: teams upgrade the dependency (bundle), they don't rewrite application code (architecture). Platform upgrades propagate to 100% of teams in one release cycle.

## Files

| Path | Description |
|------|-------------|
| `calm/trades-api-and-mcp-conforming.architecture.json` | Architecture (unchanged between beats) |
| `bundle-v1/` | Original deployment templates (no resource limits) |
| `bundle-v2/` | Updated deployment templates (resource limits added) |
| `generated/` | Output manifests (regenerated by demo, gitignored) |

## Cleanup

```bash
kubectl delete -k generated/kubernetes
rm -rf generated/
```
